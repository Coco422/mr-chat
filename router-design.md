**这个路由算法的核心思想是：**

**“最高优先级优先 + 透明 Failover + 按服务商独立冷却的黑名单机制”**  
（英文叫：**Priority-based Failover with Per-Provider Cooldown Blacklist**）

它专门为“多个服务商有不同额度重置周期”的场景设计，比普通负载均衡聪明得多。

### 1. 整体思路（一句话概括）
**永远先尝试最想用的服务商（优先级最高），失败了就悄悄切换到下一个，直到成功为止；同时对每个服务商单独“记仇”，连续失败几次就把它拉黑一段时间（正好等于它的额度重置周期），时间到了自动放出来继续用。**

这样既保证了**最高性价比**（尽量用 A → B），又保证了**高可用性**（C 兜底），还实现了**自动恢复**（A 额度重置后自动切回 A）。

### 2. 三大核心机制（算法的灵魂）

#### ① 优先级有序列表（Priority List）
- 把服务商排成一个**固定顺序**的 slice：`[A, B, C]`
- 这个顺序就是用户拖拽调整的结果（sort_order）
- **每次请求都从头开始遍历**（不是记住上次用哪个）
- 好处：A 恢复后，下一次请求立刻回到 A，不需要手动干预

#### ② 透明 Failover（无缝切换）
- 对外（CLI）永远只返回一次结果
- 内部：for 循环依次尝试
  - 成功 → 立刻返回
  - 失败 → 记录失败 + 可能拉黑 → **继续循环尝试下一个**
- 用户完全感觉不到中间失败了多少次

#### ③ 每个服务商独立的动态黑名单（Per-Provider Blacklist / 简化版 Circuit Breaker）
这是最精髓的部分：

- **不是全局黑名单**，而是每个服务商（A、B、C）**独立记账**
- **连续失败 N 次**（或遇到 quota 错误）→ 立即拉黑 M 分钟
  - A：N=1，M=240分钟（正好等额度重置）
  - B：N=2，M=540分钟
  - C：M=0（永不拉黑，当兜底）
- 拉黑用 **过期时间**（time.Time）记录
- 检查时：`if 当前时间 < 拉黑到期时间` 就跳过
- **自动过期 = 自动恢复**（不需要定时任务清理）

### 3. 算法执行流程（伪代码版，超清晰）

```go
for each 服务商 in 优先级列表 {   // 从 A → B → C
    if 这个服务商正在被拉黑 {
        continue  // 跳过
    }
    
    尝试调用()
    
    if 调用成功 {
        清零失败计数
        返回结果
    } else {
        失败次数 +1
        if 失败次数 >= 阈值 {
            设置拉黑到期时间 = now + M分钟
            清零失败计数
        }
        // 继续尝试下一个服务商（Failover）
    }
}
```

### 4. 为什么这个思想特别适合你的场景？

- **额度重置型服务商**（A每4小时、B每9小时）：M 直接设成重置周期，完美自动轮转
- **按量计费兜底**（C）：M=0，几乎永远不会被拉黑
- **性价比最高**：99% 的流量走 A/B，只有它们都黑了才走 C
- **并发安全**：用 Mutex 保护 map（Go 经典做法）
- **零外部依赖**：纯内存 + 时间戳就能跑（生产再加 Redis TTL 更优雅）

### 5. 这其实是行业经典模式的“定制版”

这个算法本质上是把以下三个经典模式结合在一起：
- **负载均衡的 Priority + Fallback**（Nginx upstream、Envoy priority）
- **Circuit Breaker（熔断器）**（Hystrix / Resilience4j 的半开状态思想）
- **Cooldown / Backoff**（失败后冷却一段时间再试）

区别在于：
- 传统熔断器是“全局”或“单个实例”熔断
- 你的版本是**每个服务商独立熔断 + 可配置冷却时间**（专为多供应商额度场景优化）

这就是为什么它既简单（代码不到 100 行核心），又特别“聪明”。

总结一句话：
**“用优先级保证性价比，用 Failover 保证可用性，用独立黑名单保证自动按额度周期轮转”** —— 这就是整个路由算法的思想精髓。
