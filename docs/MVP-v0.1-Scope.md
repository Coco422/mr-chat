# MrChat MVP v0.1 上线范围（KISS）

- 目标：快速上线一个“可用的聚合对话站”，核心是 **用户中心 + Chat**，其余能力在不影响安全的前提下尽量简化。
- 前提：前端为 **Vue 3 + Vite**（见 `docs/adr/ADR-0001-frontend-framework-vue.md`），Chat UI 为自研（见 `docs/adr/ADR-0002-chat-ui-strategy-self-built.md`）。

## 1. MVP 必做模块

### 1.1 Chat 模块（前端 + 后端）

前端能力：

- 会话列表：新建、重命名、删除（软删）、最近会话排序
- 消息区：发送、流式输出、停止生成、失败提示与重试
- 模型选择：从后端拉取可用模型列表（按用户/组权限过滤）
- 基础 Markdown 渲染：GFM + code block（先不做重型 mermaid/html preview）

后端能力：

- 会话/消息 CRUD（分页，按时间排序）
- SSE 流式接口（可取消）
- 最小化消息结构：`role` + `content` + `status` + `created_at`，预留 `reasoning_content` 字段
- 导出（可选）：按会话导出 Markdown（先做后端生成文件/内容即可）

### 1.2 ChatLLMRouter（网关内路由与 Failover）

目标：即使上游是自部署网关或多个 provider，也能在系统内部稳定切换与恢复。

功能范围：

- 上游（Provider/Upstream）管理：名称、base_url、鉴权方式、额度重置周期/冷却参数
- 路由策略：按“模型/用户组/租户”绑定优先级列表（priority list）
- Failover：按优先级尝试，失败自动切下一个
- 黑名单/冷却：按服务商独立记录失败计数与 `blacklist_until`（参考 `router-design.md`）
- 流式转发：支持 SSE 透传与 stop/cancel（客户端停止应打断上游连接）
- 观测：每次路由选择、失败原因、切换次数必须可追踪（日志/审计/指标至少其一）

实现建议（MVP）：

- 单机先用内存记录黑名单与失败计数
- 多实例时把状态迁移到 Redis（key 带 TTL），避免实例间不一致

### 1.3 用户中心（登录、注册、资料、配额与用量、邀请码）

登录与账户：

- 注册/登录（邮箱或用户名 + 密码）
- 密码安全：bcrypt/argon2 哈希，登录失败次数限制
- 用户资料：display_name、头像（可选）、偏好设置（JSON）

配额与用量（MVP 口径先简化）：

- 余额/额度：一个统一字段（例如 quota），按“Token 计费”先跑通
- 用量展示：至少展示总消耗与最近 N 天（或本月）消耗

邀请码（分销前置）：

- 用户拥有 `aff_code`（邀请码），可用于注册绑定 inviter
- 记录 inviter_id 与邀请人数（或邀请列表）

### 1.4 计费与兑换码系统（充值 + 防碰撞）

计费（最小闭环）：

- 预扣 + 结算（流式完成后结算，失败/取消做退款或部分结算策略）
- 消费日志：按用户、会话、模型记录 usage 与 quota 变化
- 定价：按模型配置倍率（或单价），先支持最小字段即可

兑换码充值：

- 兑换码创建（管理员）：面额、有效期、是否一次性、批次号
- 兑换码兑换（用户）：兑换成功增加 quota，落消费/充值日志
- 幂等：同一兑换码不可重复兑换（一次性码），并发兑换必须原子化

防碰撞与安全（必须做）：

- 兑换码足够长且不可预测（禁止自增/短码）
- 数据库存储只存哈希（例如 `sha256(code + server_salt)`），不存明文
- 兑换接口必须登录态，并对 IP/用户做速率限制与失败计数封禁
- 记录兑换尝试日志（成功/失败原因），便于风控与追溯
- 返回错误信息要“模糊化”，避免泄露码是否存在（统一提示：无效或已使用）

## 2. MVP 管理端（最小化）

为了能运营与排障，至少需要一个“Root/Admin”入口（UI 或 CLI 均可，但要可用）：

- 上游 Provider/Upstream 配置与启停
- 模型配置与定价倍率
- 用户查询与手动调整 quota（应有审计日志）
- 批量生成兑换码与查看兑换记录

## 3. MVP 非目标（明确不做）

- RAG、插件系统、复杂 Agent 工作流
- 支付（Stripe/支付宝/EPay 等）全链路接入
- 多租户 Workspace 级隔离（如无明确需求先不做）
- 复杂协作（多用户同会话实时编辑、在线状态等）

## 4. API 与数据模型（落地检查清单）

建议优先做 3 组 API：

1. Auth 与用户：注册、登录、当前用户、用量与 quota、邀请码信息
2. Chat：会话 CRUD、消息分页、SSE 发送与停止
3. Billing：兑换码兑换、充值/消费流水查询

数据表（MVP 最小集合）：

- users / auths（或合并也行，但建议分离）
- api_keys（可选：如果要对外提供 OpenAI 兼容 API）
- conversations / messages
- models（定价与可用性）
- logs（消费/充值流水）
- redeem_codes + redeem_redemptions（或一张表也行，但要支持审计与幂等）

## 5. 上线前“必须能跑通”的端到端路径

1. 注册 -> 登录 -> 进入 Chat
2. 兑换码充值 -> quota 增加 -> 用量页可见
3. 发起一次 SSE 流式对话 -> 停止生成 -> 正常结算/退款 -> 流水可追溯
4. 上游 provider 故障 -> Router 自动切换 -> 恢复后自动回到高优先级 provider
