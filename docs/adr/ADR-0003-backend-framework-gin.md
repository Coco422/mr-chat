# ADR-0003: 后端框架选型为 Gin

## 背景与问题

MrChat 的后端承担统一业务入口、鉴权、SSE 流式代理、路由决策、额度结算与审计记录。当前文档已经收敛到“Go 作为唯一业务入口”，但 Web 框架仍未正式落成 ADR。

我们的核心约束：

- 流式链路必须稳定，易于处理中途断连与取消
- 中间件能力要够成熟，方便接入 `request_id`、鉴权、日志、限流与错误收敛
- 学习与接入成本不能过高，避免在基础设施层消耗过多时间

## 选项

### 选项 A：Gin

- 优点：
  - Go 生态中成熟度高，社区与资料丰富
  - 中间件模型直接，适合组织鉴权、日志、错误处理与请求上下文
  - 对 SSE、流式写回、连接中止检测的实现路径清晰
- 缺点：
  - 相比更轻量的路由方案，抽象稍重
  - 需要在项目层自己约束 handler/service/repository 分层

### 选项 B：Chi

- 优点：
  - 更贴近标准库，路由层轻
  - 对喜欢显式组合的团队更灵活
- 缺点：
  - 需要自己补更多约定与脚手架
  - 对当前阶段的交付速度帮助不如直接采用成熟框架

### 选项 C：Echo / Fiber

- 优点：
  - 都能快速搭项目，功能也完整
- 缺点：
  - 团队现有参考与沉淀更多集中在 Gin
  - 迁移心智与样板积累不如直接选 Gin

## 决策

选择 **Gin** 作为 MrChat 的后端 Web 框架。

理由：

- 它足够成熟，能降低登录鉴权、SSE 流式、日志链路和错误处理中间件的落地成本
- 与当前已有 Go 方向和既有参考资料保持一致，团队切入成本低
- 在 v0.1 阶段，我们更需要稳定和交付速度，而不是追求极简框架差异

## 影响

- 后端工程骨架、路由分组、中间件与错误收敛统一按 Gin 组织
- SSE 接口直接由 Gin handler 暴露，前端直连 Go 服务
- 项目应尽早建立清晰目录分层，避免业务逻辑堆积在 handler 中

## 后续行动

1. 在任务清单中把后端骨架明确为 Gin。
2. 在实现时优先落地通用中间件：`request_id`、恢复、鉴权、审计上下文、限流。
3. 保持业务层与框架解耦，避免把核心路由/结算逻辑直接写死在 Gin handler 中。
