## 背景与问题

MrChat v0.1 需要承载用户中心、会话消息、额度账本、兑换码、审计日志和模型路由配置。此前文档里数据库长期停留在 “MySQL / PostgreSQL 二选一” 的状态，需要正式收敛，避免迁移、索引和本地开发方案继续摇摆。

我们的核心约束：

- 账本、兑换与调额操作必须依赖可靠事务与行级锁
- 会话、消息、审计、配置里存在较多 JSON 扩展字段
- 后续需要较灵活的查询、统计和索引能力，而不想过早受限

## 选项

### 选项 A：PostgreSQL

- 优点：
  - 事务、行级锁和并发控制能力强，适合 `quota_logs`、兑换码、调额等强一致路径
  - `jsonb`、表达式索引、部分索引等能力更适合当前数据模型
  - 对审计、用量聚合、配置扩展字段的演进更友好
- 缺点：
  - 团队若更习惯 MySQL，需要补一部分运维与 SQL 习惯切换

### 选项 B：MySQL

- 优点：
  - 团队与外部参考项目里已有较多现成经验
  - 生态成熟，常规 CRUD 没问题
- 缺点：
  - 对当前文档里大量 JSON 扩展、复杂索引和后续分析查询的适配性略弱
  - 继续保留双选项只会让迁移脚本和实现细节长期悬空

## 决策

选择 **PostgreSQL** 作为 MrChat 的主数据库。

理由：

- 它更适合当前“账本 + 审计 + JSON 扩展字段 + 后续统计查询”的组合场景
- 能更自然支撑兑换码幂等、额度预扣/退款、消息状态流转等事务性路径
- 统一选型后，迁移脚本、索引策略和本地开发环境都可以一次性收敛

## 影响

- 所有 v0.1 迁移脚本与索引设计以 PostgreSQL 为目标
- 本地开发默认依赖改为 PostgreSQL + Redis
- 旧文档里 MySQL 相关内容保留为背景材料，不再作为当前实现基线

## 后续行动

1. 在需求基线与任务清单中明确 PostgreSQL 已选定。
2. 尽快补一份实现约束：PostgreSQL 版本、迁移工具、连接池参数与本地初始化方式。
3. 在数据层实现中优先把 UUID、JSON 字段、索引策略一次性按 PostgreSQL 设计到位。
