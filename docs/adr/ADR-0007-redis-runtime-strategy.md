# ADR-0007: Redis 仅作运行时加速层，并要求故障可降级

## 背景与问题

MrChat 需要考虑未来多实例扩容，也需要在早期阶段保持系统简单可控。Redis 适合承担缓存、限流和共享短状态，但如果边界不清，很容易被误用成业务真相源，导致 Redis 故障时整站不可用。

当前约束：

- 额度账本、兑换码、消息和审计都属于强一致业务数据
- 登录安全、上游冷却、热配置缓存又天然适合放在 Redis
- v0.1 不希望为了未来扩容过早引入复杂分布式架构，但必须预留正确方向

## 选项

### 选项 A：Redis 作为加速层与共享短状态层

- 优点：
  - 与 PostgreSQL 分工清晰，业务真相与运行时状态边界明确
  - 早期实现简单，后期横向扩容也平滑
  - Redis 故障时更容易降级，而不是拖垮核心链路
- 缺点：
  - 需要为每类 Redis 用途定义 TTL、回源、降级和告警规则
- 风险与未知：
  - 如果抽象层做得不好，业务代码仍可能偷偷依赖 Redis 真相
- 迁移/回滚成本：
  - 低，符合当前阶段

### 选项 B：Redis 承担部分业务真相

- 优点：
  - 某些路径看上去更快
  - 单机压测时实现可能更直接
- 缺点：
  - 强一致路径复杂度显著上升
  - Redis 故障会直接影响余额、兑换、消息或审计正确性
- 风险与未知：
  - 运维和故障恢复成本高，极易产生数据不一致
- 迁移/回滚成本：
  - 高

### 选项 C：早期完全不用 Redis

- 优点：
  - 架构最简单，依赖最少
  - 所有状态都在 PostgreSQL，心智统一
- 缺点：
  - 限流、冷却和共享短状态缺少合适载体
  - 后续多实例扩容会重新补一轮运行时层设计
- 风险与未知：
  - 未来扩容时变更面更大
- 迁移/回滚成本：
  - 中

## 决策

选择 **选项 A：Redis 作为加速层与共享短状态层**。

## 影响

- PostgreSQL 继续作为用户、会话、消息、额度、兑换码、审计的真相源
- Redis 仅用于：
  - 登录失败计数与接口限流
  - 上游失败计数、冷却窗口和共享短状态
  - 热配置缓存和可重建缓存
  - 短期幂等键与轻量跨实例通知
- Redis 故障时必须满足：
  - 缓存旁路回源，不阻断核心 API
  - 限流/冷却退化为单实例内存策略
  - 登录、Chat、结算、兑换、审计链路继续可用
- 不使用 Redis 分布式锁保护账本、兑换和余额等强一致路径；这些路径依赖 PostgreSQL 事务和显式锁
- API 实例保持尽量无状态，未来扩容优先通过横向扩容应用实例实现

## 后续行动

1. 抽象 `Cache`、`RateLimiter`、`EphemeralStateStore` 三类接口，避免业务代码直接散落 Redis 细节。
2. 约定统一 key 命名空间、TTL 和回源策略。
3. 在 QA 中加入 Redis 故障降级验证。
4. 在部署文档中加入 Redis 故障告警与降级处理说明。
