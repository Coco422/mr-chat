# MrChat v0.1 统一需求基线

- 状态：工作基线（用于需求收敛与实现设计）
- 日期：2026-03-12
- 适用阶段：需求分析期 -> 实现设计期
- 口径说明：若本文与 `Plan.md`、`Architecture-Design-OUI-Integration.md`、调研笔记存在冲突，以本文为准；旧文档继续保留作为背景材料与设计来源。

## 1. 产品定义

MrChat 是一个面向团队/运营场景的聚合式 LLM 对话站。

v0.1 的目标不是做“功能最全”的 AI 平台，而是尽快上线一个同时满足以下条件的首版产品：

- 用户可注册登录并进入统一 Chat 界面
- 平台可接入多个上游并稳定路由/切换
- 用量、额度、结算、审计可以闭环
- 前端流式体验和长对话可用性从一开始就被纳入约束

一句话定义：

- v0.1 = `用户中心 + 自研 Chat + 内部 LLM 路由 + Token 额度结算 + 最小运营后台`

## 2. 本版统一后的关键结论

### 2.1 产品与技术路线

- 前端统一为 Vue 3 + Vite SPA
- 核心 Chat UI 统一为自研，不将 iframe / Portal 作为主聊天路径
- 平台允许挂载外部子服务入口，支持 iframe 嵌入或新窗口跳转
- Go 后端统一使用 Gin 作为 Web 框架，并作为唯一业务入口与网关，前端直连 Go 的 SSE 接口
- 默认 ORM 统一采用 GORM；复杂查询、显式锁和账本关键路径允许直接 SQL
- 主数据库统一采用 PostgreSQL
- 数据库迁移统一采用 goose 管理版本与执行顺序，不使用 GORM `AutoMigrate` 管生产 schema
- Redis 仅用于缓存、限流和共享短状态；Redis 故障时系统必须可降级运行，不能成为账本和核心请求链路的单点真相源
- 对话流式统一采用 SSE；WebSocket 不作为 v0.1 必做项
- 对外与对内的聊天协议优先向 OpenAI 兼容格式靠拢

### 2.2 范围边界

- v0.1 先做“可运营的聚合对话站”，不做重型 AI 工作台
- 支付全链路不进入 v0.1；充值闭环用“兑换码 + 管理员手动调额”完成
- 多租户先做到“用户组/权限/模型可见性”，不做 Workspace 级独立空间隔离
- 首批上游以 OpenAI 兼容接口为主；原生 Anthropic / Gemini 适配器不作为首发阻塞项
- 外部子服务入口属于补充能力，不替代 MrChat 自身的用户中心与主 Chat

### 2.3 统一数据口径

- 业务主键统一使用 UUID 字符串
- 额度口径统一为 Token 额度
- 计费口径统一为“预扣 + 最终结算”
- 消息结构至少支持 `content` 与 `reasoning_content`

## 3. 角色与核心场景

### 3.1 角色

- `Root`：系统级管理者，可管理上游、模型、用户、额度与审计
- `Admin`：运营管理者，可管理用户、兑换码、模型可见性、额度调整
- `User`：普通使用者，可聊天、查看用量、维护个人资料、兑换充值码

### 3.2 核心场景

- 普通用户注册登录后，选择可用模型并进行流式对话
- 用户可以查看自己的剩余额度、最近用量和消费流水
- 用户可以通过兑换码获得额度
- 用户可以从平台进入受控的外部子服务进行体验
- 管理员可以配置上游和模型，查看用户与流水，并手动调额
- 当某个上游故障或额度耗尽时，系统会自动切换到下一个可用上游

## 4. v0.1 范围

### 4.1 P0：首发必须具备

#### A. 认证与用户中心

- 用户注册、登录、退出、刷新会话
- 用户资料查看与编辑：`display_name`、头像、偏好设置
- 用户额度与用量查看：剩余额度、累计消耗、最近一段时间的消耗趋势
- 登录安全：密码哈希、失败次数限制、基础风控日志

#### B. Chat 核心

- 会话列表：新建、重命名、软删除、按最近时间排序
- 消息区：发送、流式输出、停止生成、失败重试
- 模型选择：仅展示当前用户或用户组可用模型
- 基础 Markdown 渲染：GFM + code block
- 最小消息结构：
  - `role`
  - `content`
  - `reasoning_content`
  - `status`
  - `created_at`
  - `metadata`（JSON，可选）

#### C. 路由与上游管理

- 上游 Provider / Upstream 管理：名称、Base URL、鉴权信息、状态
- 模型配置：模型标识、显示名、上游映射、定价、上下文长度、可见组
- 路由策略：按“模型 + 用户组”选择优先级列表
- Failover：按优先级依次尝试，失败自动切下一个
- 上游冷却：按服务商独立记录失败次数与冷却截止时间
- 路由可观测：记录每次路由命中、失败原因、切换次数

#### D. 额度、计费与充值

- 额度字段统一为 `quota`
- 首发只支持 Token 计费
- 结算规则统一为“预扣 + 最终结算”
- 记录消费流水：用户、会话、模型、usage、quota 变化
- 支持兑换码充值
- 支持管理员手动调额

#### E. 最小管理后台

- 上游配置与启停
- 模型配置与定价倍率
- 用户查询与额度调整
- 兑换码批量生成与兑换记录查看
- 审计日志与基础系统日志查看

### 4.2 P1：建议保留，但不阻塞首发

- 会话导出 Markdown
- 邀请码绑定 inviter 的最小闭环
- 面向外部调用的 API Key 管理
- 更细的账单筛选与报表图表
- 外部子服务入口：
  - 管理员可配置名称、地址、图标、展示方式（iframe / 跳转）
  - 可按用户组控制可见性
  - 可标记为“体验服务”或“独立子服务”

### 4.3 明确不做

- RAG、插件系统、复杂 Agent 工作流
- 在线支付、订单、回调、退款全链路
- Workspace 级多空间隔离
- 多人协作编辑、在线状态同步、语音/Realtime
- 重型 Markdown 能力默认开启，例如 Mermaid、HTML preview
- 对话搜索、文件夹、分享链接作为首发必做项

## 5. 统一业务规则

### 5.1 模型与上游

- v0.1 首批只要求稳定接入 OpenAI 兼容上游
- 模型是面向用户展示的逻辑实体，上游是实际请求目标
- 一个模型可以绑定多个上游，按优先级形成候选列表
- 用户侧看到的是模型，不直接感知上游切换

### 5.2 流式与停止生成

- 前端发送聊天请求时，默认走 SSE
- “停止生成”必须能打断后端到上游的请求链路
- 流式阶段只更新最后一条 assistant 消息，不重建整个消息列表
- 流式过程中不频繁写数据库；消息完成后再持久化最终内容

### 5.3 额度与结算

- 请求发出前执行预扣
- 对话完成后按实际 usage 结算
- 取消或失败时退回未使用额度
- 若上游未返回 usage，则允许使用本地估算作为回退口径，但要记录 usage 来源

### 5.4 兑换码

- 兑换码必须使用长随机串生成，数据库只存哈希
- 同一兑换码的兑换必须保证原子性与幂等
- 兑换接口必须要求登录，并有速率限制与失败记录
- 错误返回统一模糊化，避免泄露码是否存在

### 5.5 用户组与权限

- v0.1 保留 `Root / Admin / User` 三层角色
- 用户组主要用于模型可见性与后续额度策略扩展
- 用户组由管理侧维护，不要求在 v0.1 做复杂的组协作产品能力

### 5.6 外部子服务入口

- 外部子服务入口是平台补充能力，不替代主 Chat
- 支持两种进入方式：
  - iframe 嵌入
  - 新窗口或新标签页跳转
- 是否允许 iframe 由管理员按服务逐个配置，不做全局强制
- 外部服务应以“独立部署服务”看待，具备清晰的域名、鉴权和故障边界
- 若服务不适合嵌入，应优先使用跳转而非强行 iframe
- 平台侧至少要记录入口点击或访问日志，便于运营判断使用情况

### 5.7 Redis 与运行时状态

- Redis 只承载可丢失、可重建的运行时状态，不存储额度账本、兑换结果、消息真相或审计真相
- v0.1 推荐 Redis 只用于：
  - 登录失败计数与接口限流
  - 上游失败计数、冷却窗口和短时共享状态
  - 热配置与可重建缓存
  - 短期幂等键或跨实例轻量通知
- Redis 中的所有 key 都应具备明确 TTL 与命名空间
- Redis 故障时：
  - 缓存直接旁路，回源 PostgreSQL 或内存重算
  - 限流与冷却可退化为单实例内存策略
  - 核心链路如登录、会话、聊天、结算、兑换、审计仍必须可用
- 不使用 Redis 分布式锁保护 `quota_logs`、兑换码兑换、余额结算等强一致路径；这些路径以 PostgreSQL 事务与显式锁为准
- 应用实例应尽量保持无状态，未来横向扩容时通过 PostgreSQL + Redis 共享必要状态，而不是依赖本地内存真相

## 6. 统一数据基线

### 6.1 主键与时间字段

- 所有核心业务表统一使用 UUID 字符串主键
- 时间字段统一使用 UTC 时间戳或统一时间类型，不混用多种口径
- 删除操作优先使用软删除，至少对会话与消息如此

- `users`
- `auths`
- `groups`
- `group_members`
- `upstreams`

- `conversations`

- `quota_logs`

## 8. 非功能要求

### 8.1 性能

- 首屏可交互目标：中端设备与普通网络下尽量控制在 1.5s 左右
- 首 token 目标：用户发送后尽快看到反馈，前端不能成为主要瓶颈
- 长对话可用性：1000+ 条消息场景下仍应可滚动、可定位、可继续对话
- 流式渲染要做节流与批处理，避免高频全树更新

### 8.2 安全

- 密码必须安全哈希存储
- JWT 或等价会话方案必须具备刷新与失效能力
- 关键管理操作必须进入审计日志
- 兑换码、调额、模型配置、上游配置都需要审计
- 不直接复用受许可证约束的第三方代码作为首发实现前提

### 8.3 可观测

- 至少记录请求日志、路由日志、消费流水、审计日志
- 能追踪一次聊天请求使用了哪个模型、命中了哪个上游、是否发生 Failover
- 关键异常要能区分：上游错误、解析错误、额度不足、权限不足、客户端取消

### 8.4 可用性与扩容

- Redis 故障应被视为“性能/协同能力下降”，而不是整站不可用
- 任一依赖 Redis 的模块都必须定义降级策略、超时和告警
- 核心业务实例应保持无状态，可通过增加 API 实例横向扩容
- 扩容时优先遵循：
  - PostgreSQL 保存业务真相
  - Redis 保存共享短状态
  - 应用层不依赖单机内存维持跨实例一致性

## 9. 上线验收路径

以下链路全部跑通，才能视为 v0.1 满足首发条件：

1. 注册 -> 登录 -> 进入 Chat -> 获取模型列表
2. 发送一次 SSE 流式对话 -> 停止生成 -> 正常结算或退款 -> 流水可查
3. 兑换充值码 -> `quota` 增加 -> 用量页可见
4. 管理员禁用某个上游或上游故障 -> Router 自动切换 -> 日志可追踪
5. 管理员手动调整用户额度 -> 用户侧可见 -> 审计日志可追踪
