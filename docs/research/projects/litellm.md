## 0. 已知信息（先沉淀）

- 形态：
  - Python SDK（把多家 LLM 统一成 OpenAI 风格调用）
  - Proxy/Gateway（`litellm proxy`）提供统一入口与管理能力
- Proxy 常见能力（文档语义）：
  - 虚拟 key（virtual keys）
  - 路由/回退（request routing / fallbacks）
  - 缓存、限流
  - spend tracking（成本追踪）
  - 带 UI 的管理面板
- 性能宣传（需二次验证）：官网有 “8ms P95 latency at 1k RPS” 一类描述，需明确测试口径（是否仅代理开销、是否不含上游推理耗时）

## 1. 为什么要看

MrChat 如果要做“聚合镜像站”，迟早会面对：

- 多上游、多协议、多鉴权方式
- 预算/配额/成本归因
- 高并发下的稳定性（重试、熔断、限流、排队、降级）
- 可观测（指标、追踪、日志、成本报表）

litellm 在这些方面是非常典型的参考对象。

## 2. 我们关心的调研问题（按优先级）

- 性能：高并发下代理链路的开销来源（序列化、HTTP client、流式转发、队列）
- 路由：负载均衡策略、重试/退避、熔断、健康检查
- 计费：usage 的来源与一致性（不同 provider 的 token 口径差异如何统一）
- 安全：虚拟 key、租户隔离、速率限制、IP/域名策略
- 可观测：指标体系、成本追踪粒度、异常定位路径
- 与 UI 的协作：OpenAI 兼容性边界、扩展字段（reasoning/tool calls）如何透传
