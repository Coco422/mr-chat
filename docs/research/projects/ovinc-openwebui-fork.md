## 1. 为什么要看

MrChat 如果目标是“可运营的聚合镜像站”，计费与支付迟早要落地。OVINC fork 的价值在于：

- 在 OpenWebUI 的对话体验基础上补齐：充值、支付、消费报表
- 可能有更贴近国内环境的支付与运营模块（支付宝/易支付等）

## 2. 我们关心的调研问题（按优先级）

- 计费口径：
  - token 计费 vs 次数计费 vs 时长计费的产品呈现与技术实现
  - usage 来源与一致性（不同 provider 的差异怎么处理）
- 支付链路：
  - 订单状态机（pending/completed/failed/refund）
  - 回调校验、安全边界、幂等处理
- 运营报表：
  - 用户侧账单可视化
  - 管理员侧统计（按用户/组/模型/时间）
- 与对话体验的耦合：
  - 余额不足时的 UX（提示、引导、降级模型）
  - 预扣/结算对流式的影响

## 3. 待补全的动作

- 拉取源码后输出：
  - 最小可用计费闭环（MVP）拆解
  - 支付模块的接口边界与表结构建议
  - 我们如何在 MrChat 里用更少耦合实现同等能力

## 4. 对 MrChat 的直接启发（第一批）

- 计费/支付/报表这些能力如果直接嵌进对话 UI，会导致强耦合；建议在 MrChat 里把“计费结算”做成独立域（service + tables + webhook），UI 只消费聚合后的报表 API
- 需要尽早定义“可运营”产品的最小闭环：
  - 计费口径 -> 余额/套餐 -> 支付 -> 订单 -> 对账 -> 报表 -> 风控/审计
